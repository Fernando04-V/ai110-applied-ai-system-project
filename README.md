# Applied AI System: Glitchy Guesser + AI Reliability Coach

##  Original Project

This project extends **Game Glitch Investigator** from Module 1 of AI110.

The original project was a Streamlit number-guessing game intentionally filled with bugs — wrong hints, resetting secret numbers, and broken scoring. The goal was to identify, document, and fix those bugs using AI-assisted debugging tools like GitHub Copilot. Core logic was refactored from `app.py` into `logic_utils.py`, and automated tests were added using `pytest`.

---

##  Summary

This extended system adds an **AI-powered Reliability Coach** on top of the repaired guessing game. Players can now request strategic tips generated by Claude mid-game, receive a **strategy confidence score** based on how well they're narrowing the number range, and every guess is automatically logged to a local file for traceability.

The goal is to demonstrate a complete applied AI system: a working application, an integrated AI feature, automated testing, and clear logging/guardrails.

##  Design Decisions

**Why a Reliability/Testing System?**
It was the most natural extension of a debugging-focused project. The original Module 1 was literally about fixing unreliable AI-generated code — adding a system that *measures and improves* reliability is a direct evolution of that theme.

**Tradeoffs:**
- The confidence score is a heuristic (midpoint distance), not a true measure of optimal play. A player could score low even with a good strategy in edge cases.
- Logging goes to a local flat file (`game_log.txt`) rather than a database. This is simple and readable but doesn't scale.

##  Testing Summary

Run with: `pytest -v`

| Test File | Tests | Result |
|---|---|---|
| `test_game_logic.py` | 25 | ✅ All pass |
| `test_evaluator.py` | 12 | ✅ All pass (API mocked) |

**What was tested:**
- `check_guess` covers win, too high, too low, and boundary cases
- `parse_guess` covers empty input, None, non-numeric, decimals, out-of-range
- `update_score` covers win (early/late), loss, and unknown outcomes
- `evaluate_confidence` covers no history, single guess, binary search, random guesses
- `evaluate_guess_history` covers successful API call, timeout, connection error, and malformed response

**What was learned:**
- Mocking with `unittest.mock.patch` is essential for testing any function that calls an external API — without it, tests would be slow, flaky, and dependent on an internet connection.
- The `evaluate_confidence` function originally had a divide-by-zero bug when `max_distance` was 0 (equal range edges). A guard clause fixed this.

##  Reflection and Ethics

**Limitations and biases:**
The AI coach's tips are only as good as the prompt. If the guess history is very short (1-2 guesses), Claude doesn't have enough pattern data and gives generic advice. The confidence score is also biased toward binary search — a player using a different valid strategy (e.g., always guessing near boundaries) will score low even if their logic is sound.

**Misuse potential:**
The game itself is low-stakes. However, the pattern of "send user behavior history to an AI for analysis" could be misused in a more sensitive application. Guardrails here include: the history only contains numbers (no PII), the API is called only on explicit user action (button press), and all data is logged locally rather than sent anywhere else.

**Surprises during testing:**
The AI coach was surprisingly good at detecting binary search patterns even with just 3-4 guesses. It was also surprisingly bad when given a history of only 1 guess — it would sometimes suggest the exact same number already guessed, which is useless.

**AI Collaboration:**
- **Helpful suggestion:** When writing `evaluate_confidence`, Claude suggested using the distance from the midpoint as a proxy for binary search quality — a clean heuristic I hadn't thought of.
- **Flawed suggestion:** Claude initially suggested storing the API key directly in `app.py` as a string. This is a security anti-pattern. The fix was to rely on the platform-level key injection instead of hardcoding it.
